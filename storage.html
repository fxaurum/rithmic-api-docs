TickStore — chia theo phiênTickStore — partitioned by session

  • .tick — header 32 byte (magic FXTK, version 2, recSize 29) rồi record 29 byte: time(int64 ms,8) · price(double,8) · qty(int32,4) · side(byte,1) · tu(int64,8). tu = mốc khớp sàn (µs), nằm thẳng trong record. Gom cumulative trade kiểu ATAS theo tu giống nhau. Dedup live-write chỉ bỏ record lặp NGAY trước trùng cả (ms,price,qty,side,tu) → chặn Image-snapshot lúc (re)subscribe mà KHÔNG bỏ 2 trade thật cùng ms/price/qty/side (khác tu) → không hụt volume. .tick — a 32-byte header (magic FXTK, version 2, recSize 29) then 29-byte records: time(int64 ms,8) · price(double,8) · qty(int32,4) · side(byte,1) · tu(int64,8). tu = exchange match time (µs), in-record. Group cumulative trades like ATAS by identical tu. Live-write dedup drops only the immediately-repeated record matching on all of (ms,price,qty,side,tu) → suppresses the Image-snapshot resend on (re)subscribe WITHOUT dropping two genuine same-ms/price/qty/side trades (different tu) → no volume undercount.
  • tick.coverage — 1 file text/mã: dòng <tag> <fullUpToMs> <flag>, flag D=có data · V=đã xác minh tới hết phiên (grab sạch chạm cuối phiên — phiên đóng sớm/lễ khỏi re-sync) · E=rỗng-đã-kiểm · P=pre-liquid; thêm dòng FINAL <tag> = hợp đồng đã chốt → khỏi sync lại. Không hạ cấp (V>D>E/P). tick.coverage — one text index per symbol: lines <tag> <fullUpToMs> <flag>, flag D=has data · V=verified to session end (a clean grab reached the end — an early-close/holiday half-day isn't re-synced) · E=checked-empty · P=pre-liquid; plus a FINAL <tag> line = contract fully captured → never synced again. No downgrade (V>D>E/P).
  • 1 file = 1 phiên. DayKey(ms) = (giờ Central − 17:00).ngày = trade-date CME (roll 17:00 CT, theo DST). Read trải qua các file phiên trong khoảng; phiên không cắt giữa nửa đêm UTC. 1 file = 1 session. DayKey(ms) = (Central − 17:00).date = CME trade-date (roll 17:00 CT, DST-aware). Read spans the session files in range; sessions never split at UTC midnight.

Đổi khung thời gian KHÔNG đụng tick store. Tick là thô; 1m/5m/1h chỉ đổi cách gộp để hiển thị. Đổi khung -> chỉ ĐỌC lại kho, không ghi đè. Changing the timeframe does NOT touch the tick store. Ticks are raw; 1m/5m/1h only changes aggregation. A timeframe change re-reads the store, never rewrites it.

Các lớp cacheCaching layers

  • Big-trade .bigcache (mỗi phiên đã đóng). bigTrades gom tick server-side (theo tu) ra bóng [t, side, vol, p0, p]. Phiên đóng -> đọc/ghi big_<tag>_<min>.bigcache (tức thì). Phiên hiện tại tính live. Cache hợp lệ nếu .bigcache mới hơn .tick của phiên. Tham số merge (cửa sổ cumulative kiểu ATAS) bỏ qua cache, tính tươi. Big-trade .bigcache (per closed session). bigTrades groups ticks server-side (by tu) into bubbles [t, side, vol, p0, p]. Closed sessions read/write big_<tag>_<min>.bigcache (instant); the current session is computed live. Cache valid only if the .bigcache is newer than the session's .tick. The merge param (ATAS cumulative window) bypasses the cache, computed fresh.
  • Kho nến (.bars+.coverage). Klines cache theo khung; phiên ĐÓNG có tick → dựng nến 1m TỪ tick (verify khớp byte-for-byte với klines Rithmic), gắn cờ t(tick)/k(kline). m5/15m/1h/4h gộp từ 1m. → mở lại/đổi khung đọc local, khỏi gọi Rithmic. Bar store (.bars+.coverage). Klines cached per interval; a closed session with ticks builds its 1m candles FROM ticks (verified byte-for-byte equal to Rithmic klines), tagged t(tick)/k(kline). m5/15m/1h/4h aggregate from 1m. → reopen/timeframe-switch reads locally, no Rithmic call.
  • Cache symbol. Search trả cache-first tức thì; nền refresh 1 lần/phiên. Search ES cache luôn MES* -> gõ MES sau là hit. Lọc bỏ spread (-) / option (dấu cách). Symbol cache. Search returns cache-first instantly; a once-per-session background refresh tops it up. Searching ES also caches MES* -> typing MES later is a hit. Spreads (-) / options (space) filtered out.
  • Cache trade (RAM). loadTrades giữ trade ngày live trong RAM; footprint/DOM/big-trade seed từ đây thay vì mỗi cái replay riêng. Trade cache (RAM). loadTrades keeps the live day in memory; footprint/DOM/big-trade seed from it instead of each replaying separately.

Search — chỉ 1 lúc 1 cáiSearch — single-in-flight

Rithmic chỉ cho 1 searchInstrument tại một thời điểm, và không có API liệt kê toàn bộ. Vì vậy: Rithmic allows one searchInstrument at a time and has no list-all API. So:

  • Gateway xếp hàng search: 1 cái chạy, còn lại chờ; gõ nhanh chỉ giữ query mới nhất.The gateway serializes searches: one active, rest queued; fast typing coalesces to the latest.
  • Gặp in use -> chờ 400 ms thử lại (≤ 12 lần) thay vì fail.On in use -> wait 400 ms and retry (≤ 12×) instead of failing.
  • Web app chọn mã từ cây product (bung 1 product = 1 search nhắm đúng) + ô tìm — KHÔNG preload-tất-cả. Đó là lý do không có "load hết lúc connect".The web app picks from a product tree (expand a product = one targeted search) + a search box — never a bulk preload. Hence no "load-all-on-connect".

@depth<N> — độ sâu theo subscriptionper-subscription depth

Sổ lệnh giữ tối đa DOM_MAX_LEVELS = 1000 mức/bên. Mỗi lần phát bọc trong DepthBox; Route cắt cho từng client: @depth<N> -> N mức gần giá nhất, @depth trơn -> toàn bộ. Web app subscribe @depth<N> với N từ ⚙ Heatmap (mặc định 100, dùng chung DOM + heatmap) -> chỉ kéo sâu đúng mức đang xem. MBP của Rithmic cấp hàng trăm mức — không cần MBO. The book holds up to DOM_MAX_LEVELS = 1000 levels/side. Each emit is wrapped in a DepthBox; Route serves each subscriber its own slice: @depth<N> -> the N nearest-to-price levels, plain @depth -> all. The web app subscribes @depth<N> with N from the Heatmap ⚙ (default 100, shared by DOM + heatmap) -> you pull only as deep as you view. Rithmic MBP supplies hundreds of levels — no MBO needed.

Thứ tự subscribe quan trọng: subscribe trong lúc replay sẽ huỷ replay -> subscribe LIVE trước, replay nền sau. 2 nguồn replay: replayTrades = CHỈ phiên hiện tại (đuôi live / footprint hôm nay); replayBars(Tick,1) = lịch sử tick sâu, replay được phiên ĐÃ ĐÓNG (tới ~lúc hợp đồng niêm yết, nhưng chỉ khi CÒN niêm yết — hết hạn/delist thì rỗng; cuối tuần/bảo trì bị từ chối rp=14). TickStore là kho lâu dài, tích luỹ từng phiên (grab xong sống sót cả khi hợp đồng delist). Subscribe order matters: subscribing during a replay aborts it -> live subscribe first, background replay after. Two replay sources: replayTrades = the CURRENT session only (today's live tail / footprint); replayBars(Tick,1) = the deep tick history that DOES replay CLOSED sessions (back to ~the contract's listing start, but only while still listed — an expired/delisted contract returns empty; weekend/maintenance rejects it with rp=14). The TickStore is the long-term archive, accumulating session by session (a grabbed session survives the contract's delisting).

Đồng bộ phiên đóng = gap-patch. SyncTicksJob grab (fullUpTo → cuối phiên] qua replayBars(Tick) rồi Merge (đè cửa sổ) — vì fullUpTo là mốc verify liền-mạch-từ-đầu, MỌI lỗ phía trên nó (tắt/mở nhiều lần trong phiên: ■■ trống ■■ trống ■■) được lấp một lượt, không chỉ đuôi. fullUpTo chỉ tiến khi replay SẠCH (RpCode=0); NeedsSync kiểm chéo tick cuối thật vs cuối phiên (biên halt ~90') → coverage nói-dối tự lành lần mở sau. Ghi file fsync + đổi tên atomic, coverage ghi sau khi data bền. Archiver chỉ mã đang mở (không hốt hợp đồng anh em; ⤓ Lịch sử để chộp prior). Subscription linger ~6s sau khi ref=0 → bớt churn sub/unsub + hết race in use. Closed-session sync = gap-patch. SyncTicksJob grabs (fullUpTo → sessionEnd] via replayBars(Tick) and Merges (window-replace) — since fullUpTo is the contiguous-verified watermark, EVERY hole above it (closing/reopening the app several times in a session: ■■ gap ■■ gap ■■) is refilled in one pass, not just the tail. fullUpTo only advances on a CLEAN replay (RpCode=0); NeedsSync cross-checks the real last tick vs session end (~90' halt margin) → an over-claimed coverage self-heals next open. Writes are fsync + atomic rename, coverage written after the data is durable. Archiver is opened-symbol-only (no sibling fan-out; ⤓ Lịch sử for prior contracts). Subscriptions linger ~6s after ref hits 0 → less sub/unsub churn + no in use race.
