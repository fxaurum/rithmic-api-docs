Heatmap là gìWhat it is

Heatmap thanh khoản theo thời gian × giá, kiểu Bookmap: mỗi cột pixel là một thời điểm, mỗi hàng là một giá; độ sáng ô = size lệnh chờ tại (thời gian, giá). Vẽ đè thêm: đường best bid/ask, bóng trade, và histogram sổ lệnh hiện tại ở mép phải. Là một primitive LWC nằm dưới nến (bật "Hiện nến" thì footprint/nến đè lên). A liquidity heatmap over time × price, modelled on Bookmap: each pixel column is a moment, each row a price; cell brightness = resting order size at that (time, price). Overlaid: a best bid/ask line, trade bubbles, and a current order-book depth histogram on the right edge. An LWC primitive drawn under the candles (turn on "show candles" to overlay footprint/candles).

Dải màu (ít → nhiều thanh khoản): tối → xanh dương → cyan → trắng → vàng → cam → đỏ. Gamma nâng tương phản. Colormap (low → high liquidity): dark → blue → cyan → white → yellow → orange → red. Gamma boosts contrast.

Hai chế độ. Mặc định thân heatmap là sổ gộp @depth lấy mẫu thành cột. Bật ⚙ → MBO để đổi thân sang thanh theo đời lệnh dựng từ DBO (@mboraw): mỗi lệnh chờ = 1 hình chữ nhật rộng = thời gian tồn tại, xếp lane khi chồng nhau tại một giá — xem §MBO. (EMA theo thời gian + phân loại iceberg/spoof vẫn nằm trong roadmap MBO.) Two modes. By default the heat body is the aggregated @depth book sampled into columns. Toggle ⚙ → MBO to switch the body to per-order lifetime bars from DBO (@mboraw): each resting order = a rectangle width = its lifetime, lane-stacked when orders overlap at a price — see §MBO. (Time-weighted EMA + iceberg/spoof classification stay on the MBO roadmap.)

Tối thiểu = @depth (ô + histogram) + @bookTicker (đường). @trade chỉ thêm bóng. Depth cần quyền L2 từ Rithmic. Minimum = @depth (cells + histogram) + @bookTicker (line). @trade only adds bubbles. Depth needs L2 entitlement.

Chế độ MBO đổi nguồn thân heatmap: thay @depth bằng @mboraw (sự kiện đời lệnh DBO → các thanh) + @mbo<N> (size từng mức → cột sổ lệnh mép phải và reconcile bên dưới) + giữ @depth1000 (BBO để dọn lệnh chéo). Stream sự kiện có thể trôi (sót New/Delete) nên các thanh per-order được đối soát với @mbo mỗi 300 ms — đúng nguồn size từng mức mà DOM hiển thị — nên bức tranh per-order của heatmap khớp DOM chính xác (xem §MBO). @bookTicker/@trade giữ nguyên. Cần quyền DBO (kiểm bằng mboProbe, xem MBO). Nút MBO và DOM dùng chung các stream này nhưng độc lập — tắt cái này vẫn giữ stream cái kia cần (MBO trên heatmap chạy được khi panel DOM đóng). MBO mode swaps the heat-body source: instead of @depth it subscribes @mboraw (per-order DBO lifetime events → the bars) + @mbo<N> (per-level sizes → the right-edge ladder and the reconcile below) + keeps @depth1000 (BBO for crossed-order cleanup). Event streams can drift (a missed New/Delete), so the per-order bars are reconciled against @mbo every 300 ms — the same authoritative per-level sizes the DOM shows — so the heatmap's per-order picture matches the DOM exactly (see §MBO). @bookTicker/@trade unchanged. Needs DBO entitlement (verify with mboProbe, see MBO). The MBO and DOM toggles share these streams but are independent — turning one off keeps whatever the other still needs (MBO on the heatmap works with the DOM panel closed).

Các lớp vẽ (dưới → trên)Layers (bottom → top)

  1. Ô nhiệt — mỗi cột 300 ms, mỗi mức sổ là 1 ô màu. Sáng = log(size+1)/log(max) mũ gamma. Cột lấp đầy từ cột trước (không hở sọc đen). Luôn trải full bề ngang pane. Heat cells — per 300 ms column, each book level is a colored rectangle. Brightness = log(size+1)/log(max) ^ gamma. Columns gap-fill (no black stripes). Always stretched to full pane width.
  2. Đường bid/ask — step line từ @bookTicker. Trái vạch vàng = nét liền; phải (vùng sổ lệnh) = nét đứt. Chưa có tick mới thì giữ giá cuối kéo ngang tới mép phải. Bid/ask line — step line from @bookTicker. Left of the yellow line = solid; right (depth zone) = dashed. Holds the last value to the right edge when no new tick.
  3. Sổ lệnh hiện tại (Bookmap mép phải) — thanh ngang của sổ mới nhất: đỏ = ask (trên giá), xanh = bid (dưới). Dài ∝ size; cột to nhất chạm "vạch vàng". Current order-book depth (Bookmap right edge) — horizontal bars of the latest book: red = ask (above), green = bid (below). Length ∝ size; the largest bar reaches the "yellow line".
  4. Bóng trade — vòng tròn mờ tại (thời gian, giá) của lệnh.Trade bubbles — translucent circles at (trade time, price).

Chế độ MBO (thanh theo đời lệnh)MBO mode (per-order lifetime bars)

Bật ⚙ → MBO. Thân heatmap đổi từ ô @depth sang một hình chữ nhật cho mỗi lệnh chờ — đọc được đời lệnh, thứ heatmap gộp không cho thấy (chỉ biết tổng mức, không biết lệnh nào). Toggle ⚙ → MBO. The heat body changes from @depth cells to one rectangle per resting order, so you read each order's lifetime — what the aggregated heatmap can't show (it only knows the level total).

  • Thanh từng lệnh — mỗi lệnh (khoá theo ExchOrdId từ @mboraw) là 1 chữ nhật ở hàng giá của nó, x = thời gian tồn tại: mép trái = lần đầu thấy (New/Image), mép phải = bị xoá (Delete) hoặc now nếu còn chờ. Lệnh 1-lot chờ 8 s = thanh dài mảnh; lệnh 50-lot nhấp nháy 200 ms = thanh ngắn dày. Còn sống = sáng, đã chết = mờ. Per-order bar — each order (keyed by ExchOrdId from @mboraw) is a rectangle at its price row, x = its life: left = first seen (New/Image), right = removed (Delete) or now if still resting. A 1-lot resting 8 s is long & thin; a 50-lot flashing 200 ms is short & fat. Alive = bright, dead = faded.
  • Khớp DOM (reconcile). @mboraw là stream sự kiện nên có thể trôi — sót New/Image → thiếu lệnh; sót Delete → lệnh ma. Mỗi 300 ms tập lệnh sống được đối soát với @mbo (chính là sizes[] từng mức mà DOM hiển thị): size có trong @mbo mà thanh chưa có → thêm (dùng size THẬT từ @mbo, không đoán/chia bừa; đời lệnh tính từ lúc reconcile), thanh sống mà @mbo không còn liệt kê → đóng. Ưu tiên khớp lệnh thật trước, nên New thật tới sau sẽ thay lệnh đắp tạm. Kết quả: per-order khớp DOM từng tick, vẫn giữ id/đời lệnh thật của @mboraw (nền cho roadmap iceberg). (@mbo cắt sizes[] ở 64/mức, nên mức hiếm có >64 lệnh chỉ đối soát 64 đầu và không đóng nhầm thanh nào.) Matches the DOM (reconcile). @mboraw is an event stream, so it can drift — a missed New/Image leaves an order out; a missed Delete leaves a phantom. Every 300 ms the live set is reconciled against @mbo (the same authoritative per-level sizes[] the DOM shows): a size in @mbo the bars lack is added (with the real @mbo size — never a guessed split; lifetime starts at reconcile time), a live bar @mbo no longer lists is closed. Real orders match first, so a late real New supersedes any stand-in. Result: the per-order picture agrees with the DOM tick-for-tick while the real @mboraw ids/lifetimes are kept (basis for the iceberg roadmap). (@mbo caps sizes[] at 64/level, so on the rare >64-order level only the top-64 are reconciled and no bar is wrongly closed.)
  • Lane — các lệnh trùng thời gian tại cùng giá xếp thành lane dọc trong hàng tick (xếp khít, lane tái dùng khi lệnh chết). Chính là "3 ô 2/3/5 tại một giá" của DOM nhưng có chiều thời gian. Lanes — orders overlapping in time at one price stack into vertical lanes (greedy packing; a lane is reused when its order dies). The DOM's "3 blocks of 2/3/5 at one price" given a time dimension.
  • Cap lệnh chết. Một giá sôi động có thể luân chuyển hàng trăm lệnh mỗi phút; giữ hết thanh chết vừa phình bộ nhớ vừa đè lane. Mỗi giá giữ 40 lệnh TO nhất (HEAT_DEADCAP), bỏ đám 1-lot noise — cắt theo CỠ, không theo thời gian, nên lịch sử lệnh to (tín hiệu) được giữ, giá vắng giữ đủ. Lệnh sống không bao giờ cắt. Dead-order cap. A busy price can churn hundreds of orders a minute; keeping every dead bar bloats memory and crowds the lanes. Per price we keep the 40 largest dead orders (HEAT_DEADCAP) and drop 1-lot noise — cutting by size, not time, so large-order history survives and quiet prices keep everything. Live/resting orders are never capped.
  • Số — mỗi thanh in cỡ lệnh khi lane đủ cao và thanh đủ rộng. Number — each bar prints its order size when the lane is tall and the bar wide enough.
  • Tô lệnh lớn — lệnh ≥ Tô màu lệnh vẽ theo màu side (bid xanh / ask đỏ, chỉnh được); còn lại trung tính trắng. Ẩn lệnh < bỏ hẳn lệnh nhỏ. Big-order highlight — orders ≥ Tô màu lệnh render in the side color (bid green / ask red, configurable); the rest neutral white. Ẩn lệnh < drops small orders entirely.
  • Dải theo tick — mỗi tick giá có nền zebra + đường border xanh xám mảnh, để mỗi hàng giá là một lane rõ chứa các thanh. Per-tick bands — every price tick gets a zebra background + a thin blue-grey border line, so each row is a clear lane the bars sit inside.
  • Mép phải — thân heatmap cắt bên trái vạch vàng; vùng phải giữ sổ lệnh live, nay vẽ từ tổng @mbo (= cột Vol của DOM nên khớp). Đường bid/ask nét liền bên trái vạch vàng, nét đứt phẳng bên phải. Right edge — the heat body is clipped left of the yellow line; the right zone keeps the live ladder, now from @mbo totals (= the DOM Vol column, so they agree). The bid/ask line is solid left of the yellow line, dashed flat to its right.
  • Trục thời gian — cửa sổ stretch scale cố định (heatViewMs, mặc định 15 s), now ghim ở vạch vàng, quá khứ trượt mượt sang trái. Zoom quanh con trỏ, kéo chuột để xem lại (xem Tương tác); nút ⟶ hiện tại / double-click về live. Time axis — a fixed-scale stretch window (heatViewMs, default 15 s), now pinned at the yellow line, history sliding smoothly left. Zoom is cursor-anchored, drag to review the past (see Interaction); the ⟶ hiện tại button / double-click return to live.

MBO và DOM độc lập. Cả hai dùng @mbo/@mboraw/@depth, nhưng mỗi nút chỉ gỡ stream riêng nó cần — tắt DOM không còn làm trắng heatmap (và ngược lại). MBO and DOM are independent. Both consume @mbo/@mboraw/@depth, but each toggle only tears down the streams it alone needs — turning the DOM off no longer blanks the heatmap (and vice-versa).

Bên trongInternals

  • Sổ lệnh là cùng DomBook với DOM (dựng từ OnBidQuote/OnAskQuote + snapshot, phát @depth<N> ~30 Hz). Client lấy mẫu mỗi HEAT_MS = 300 ms -> heatCols. The book is the same DomBook as the DOM (from OnBidQuote/OnAskQuote + snapshot, emitted @depth<N> ~30 Hz). The client samples every HEAT_MS = 300 ms -> heatCols.
  • Đường bid/ask dùng mảng riêng heatQuotes theo từng @bookTicker (mịn hơn ô 300 ms) -> bóng khớp đường. X là stretch full-width (T−viewStart)/viewMs × paneWidth (LWC không có trục X dưới-cây-nến nên vẽ tay). The bid/ask line uses a separate heatQuotes fed by each @bookTicker (finer than the 300 ms cells) -> bubbles align. X is a full-width stretch (T−viewStart)/viewMs × paneWidth (LWC has no sub-bar X axis, so it's hand-drawn).
  • Chế độ MBO nuôi trên @mboraw: client giữ mboOrders (ExchOrdId → {p, q, side, t0, t1, segs[]}) — New/Image sinh, mỗi Change mở đoạn mới (đổi 12→6 vẫn giữ đoạn 12), Delete chốt t1. Mỗi 300 ms chạy tuần tự: chốt lệnh chéo (giá bị vượt → đóng tại chỗ, mốc BBO @depth), reconcileMbo() (thêm-thiếu/đóng-thừa theo @mbo; lệnh đắp tạm mang id syn:…, tái dựng mỗi nhịp nên tự dọn), prune theo cửa sổ, cap lệnh chết (HEAT_DEADCAP, giữ cái to nhất). Renderer cắt theo dải Y nhìn (coordinateToPrice(0..paneHeight)) — cả sổ ~1375 mức nhưng màn chỉ vài chục — nên kéo/zoom Y mượt. MBO mode feeds on @mboraw: the client holds mboOrders (ExchOrdId → {p, q, side, t0, t1, segs[]}) — set on New/Image, a new segment per Change (a 12→6 resize keeps the 12), closed (t1) on Delete. Each 300 ms runs, in order: crossed-order freeze (price crossed → closed there, using the @depth BBO), reconcileMbo() (add-missing / close-phantom vs @mbo; stand-ins get a synthetic syn:… id and are re-derived each tick so they self-clean), the window prune, and the per-price dead cap (HEAT_DEADCAP, largest-kept). The renderer culls by the visible Y range (coordinateToPrice(0..paneHeight)) — the book is ~1375 levels but the screen shows a few dozen — which keeps Y drag/zoom smooth. On (re)subscribe the gateway pushes the current MboBook as Image events so a chart enabling MBO mid-session sees already-resting orders.
  • Xem STORAGE: cách @depth<N> cắt theo từng subscription để DOM + heatmap dùng chung 1 stream ở độ sâu chọn. See STORAGE: how @depth<N> is trimmed per subscription so DOM + heatmap share one stream at a chosen depth.
